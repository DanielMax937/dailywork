# 架构说明

本仓库包含 **两套彼此独立、同栈共存** 的能力：面向浏览器的 **Next.js 应用**（Todo），以及本机常驻的 **自动化 Worker**（cron 调度 + Telegram Bot）。二者共享 TypeScript / Drizzle / SQLite 技术选型，但 **进程与数据库文件分离**，避免互相牵制。

## 1. 进程与职责

| 组件 | 启动方式 | 职责 |
|------|----------|------|
| **Web 应用** | `npm run dev` / `npm run start` | App Router 页面、Server Actions；数据在 `data/sqlite.db`。 |
| **自动化 Worker** | `npm run worker` | 读取 `data/automation.db` 中的任务，注册 cron；通过 Telegraf 长轮询处理私聊命令；执行 shell 命令并写运行日志、发 Telegram 通知。 |

设计上 Worker **不嵌入** Next 生命周期（避免 `next dev` 热重载导致调度重复初始化）。本地开发通常 **两个终端** 分别启动 Web 与 Worker。

## 2. 数据层

| 数据库文件 | Drizzle 配置 | Schema | 用途 |
|------------|--------------|--------|------|
| `data/sqlite.db` | `drizzle.config.ts` | `src/db/schema.ts`（`todos`） | Todo 列表。 |
| `data/automation.db` | `drizzle.automation.config.ts` | `src/automation/db/schema.ts`（`tasks`、`run_logs`） | 定时任务定义与每次执行的日志。 |

迁移/推表：`npm run db:push` 与 `npm run automation:db:push` 分别对应上述两个库。

## 3. 自动化子系统（`src/automation/`）

- **入口**（`index.ts`）：校验 `TELEGRAM_BOT_TOKEN`、`TELEGRAM_ALLOWED_CHAT_ID`；组装 `sendMessage`；挂载 Telegram 处理器；启动调度器后 `bot.launch()`；处理 SIGINT/SIGTERM 时停止 cron 与 Bot。
- **调度**（`scheduler.ts`）：启动时查询 `enabled = true` 的任务，用 `cron` 包的 `CronJob` 注册；同一任务若上次仍在执行则跳过本次 tick（`running` 集合）；**任务变更需重启 Worker** 后生效。
- **Cron 表达式**（`cron-expr.ts`）：用户配置为 **标准五段**（分 时 日 月 周）；内部转为六段时在首位补 `0` 秒，以匹配 `cron` 库。
- **执行器**（`runner.ts`）：`execa` + `shell: true` 执行 `tasks.command`；成功判定为 **退出码 0**；`AUTOMATION_DEFAULT_TIMEOUT_SEC`、`AUTOMATION_LOG_MAX_CHARS` 控制超时与 stdout/stderr 截断；结果写入 `run_logs` 并格式化后 Telegram 通知（消息过长再截断）。
- **Telegram**（`telegram.ts`）：仅处理与 `TELEGRAM_ALLOWED_CHAT_ID` 一致的会话；命令 `/help`、`/list`、`/run <id>`。

## 4. Web 应用结构（概要）

- **`src/app/`**：页面与布局；首页为 Todo，数据通过 `@/db` 与 `@/actions/todo` 读写。
- **`src/db/`**：Todo 用的 Drizzle 客户端与 schema。

自动化逻辑 **不** 依赖 Next 服务端，仅文档与运维上与 Web 同仓库维护。

## 5. 安全与配置要点

- `.env` 存放 Telegram 与可选自动化参数，勿提交仓库。
- Bot 仅响应配置的单一私聊，降低误用风险。
